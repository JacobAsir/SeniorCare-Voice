Build a production-style full-stack web app called “SeniorCare Voice”.

High-level goal:
SeniorCare Voice is a Japan-first voice-first assistant for older adults and caregivers.
It helps users ask simple spoken questions and receive calm, accessible spoken and on-screen answers.

Core product idea:
This is not a generic chat assistant.
It is a bounded daily-life support product designed for:
- lower cognitive load
- clearer wording
- large readable UI
- voice-first interaction
- replay / repeat support
- caregiver-friendly shared use

Primary use cases for MVP:
- explain a product label or medicine label in simpler language
- explain a notice or reminder
- answer simple daily-life utility questions
- provide step-by-step guidance
- repeat or rephrase the answer more simply

Important safety boundary:
This app must not provide diagnosis, legal advice, or emergency instructions as if authoritative.
If the user asks for dangerous or high-risk advice, the app must escalate to a safe response:
- contact pharmacist
- contact doctor
- call emergency services
- ask caregiver/family support
depending on context

This version should be semi-stateless.
No auth or database unless explicitly requested later.
Use session-only in-memory context if needed, but do not persist user history.

Tech stack:
Frontend:
- Next.js 14+ with TypeScript
- Tailwind CSS
- accessible mobile/tablet-friendly UI
- microphone support
- playback controls
- large text mode

Backend:
- Python FastAPI
- speech-to-text abstraction
- response orchestration service
- text-to-speech abstraction
- session context manager (in-memory only)
- bounded intent handling
- safety and escalation rules

AI / model usage:
- use STT abstraction for voice transcription
- use Groq for bounded response generation and rephrasing
- use TTS abstraction for spoken output
- optional Gemini later for image understanding
- app must support mock mode without live providers

Architecture requirement:
Build this as a bounded voice interaction system with separate services.

Use a gateway pattern:
- frontend records or uploads audio
- backend transcribes
- backend classifies intent
- backend generates bounded response
- backend optionally returns TTS audio
- no persistent storage

Supported intents for MVP:
- explain label
- explain notice
- ask simple daily utility question
- repeat answer
- simplify answer
- caregiver handoff suggestion
- unsupported / unsafe question fallback

Required product behavior:
The user speaks or types a question.
The app:
1. transcribes audio
2. detects likely intent
3. determines whether the request is supported
4. generates a calm, simple answer
5. returns text and optional audio playback
6. offers follow-up actions such as:
   - repeat
   - simpler version
   - show text
   - caregiver mode
7. escalates if request is unsafe or too uncertain

Core MVP features:

1. Landing page
- explain what the app does

- emphasize accessibility and safety-first design
- CTA to start voice interaction

2. Talk page
- large microphone button

3. Caregiver mode

4. Safety / limitations page
- explain what the app can and cannot do
- explain escalation logic

Voice pipeline requirements:
Step 1: audio intake
- record or upload audio
- validate format
- support mock transcription for demo mode

Step 2: transcription
- transcribe to text
- return confidence if possible

Step 3: intent classification
Classify into bounded categories:
- label explanation
- notice explanation
- daily utility guidance
- repeat / simplify
- unsupported
- safety-sensitive / escalate

Step 4: response generation
- use deterministic response templates where appropriate
- use LLM only within a bounded prompt
- keep responses short, calm, and clear
- avoid overload
- expose follow-up choices

Step 5: TTS output
- optional audio synthesis
- support replay and slower pacing
- if TTS unavailable, text mode must still work
